# Edge 插件 Spec：Gemini 式 Chat 历史按需加载与虚拟化

## 1. 文档信息

**项目名**：Edge Chat Virtualizer  
**目标形态**：Microsoft Edge 扩展（Manifest V3）  
**目标网站**：ChatGPT Web（优先），后续可扩展到其他长对话型网页应用  
**版本**：Spec v0.1  
**文档目的**：定义一个以 Gemini 为交互参考的网页端长对话性能优化插件，在不改动站点后端的前提下，通过前端虚拟化与按需恢复机制，显著降低长对话页面的渲染和滚动卡顿。

---

## 2. 背景与问题定义

当前 ChatGPT 网页端在超长对话中存在明显卡顿问题，主要表现为：

- 滚动延迟明显增大
- 输入响应变慢
- 页面切换或重新聚焦时卡顿
- 长 Markdown、代码块、公式、图片等内容导致布局与重排成本升高
- 页面长期停留后，前端内存与节点规模持续增大

Gemini 网页端的用户感知行为更接近“历史按需加载”：

- 默认只保留最近少量对话内容
- 向上滚动到顶部时，再继续加载更早历史
- 用户全局搜索时，可以定位并恢复相关历史内容

本项目不追求复刻 Gemini 的后端实现，而追求复刻其**用户感知层面的使用体验**。

---

## 3. 目标

### 3.1 核心目标

1. 在长对话页面中，仅保留最近 **10 组问答记录** 的完整渲染结果。
2. 旧记录默认从 DOM 中卸载，仅保留可恢复的占位与快照。
3. 当用户**滚轮滚动到顶部**时，按批次加载更早历史记录。
4. 当用户使用**全局搜索**时，按需恢复匹配到的历史记录及其上下文。
5. 在不侵入站点后端的前提下，显著降低长对话页面的渲染负担与滚动卡顿。

### 3.2 次要目标

1. 保持用户滚动位置稳定，避免历史恢复时明显跳动。
2. 对流式生成中的回答做保护，避免生成中记录被错误卸载。
3. 提供可调参数，例如窗口大小、预加载缓冲区、批量加载条数。
4. 为后续扩展到更多站点预留抽象接口。

---

## 4. 非目标

以下内容不属于首版范围：

1. 不修改 ChatGPT 后端接口。
2. 不篡改 OpenAI 的会话存储协议。
3. 不试图接管 React 内部状态管理。
4. 不保证完全复刻站点原生所有动态交互行为。
5. 不保证站点内部 JS heap 完全等价于 Gemini 那种原生分页式回收。
6. 不做跨设备同步。
7. 不做通用网页自动化平台。

---

## 5. 关键术语

### 5.1 Turn
单条消息节点，可能是用户消息，也可能是助手消息。

### 5.2 QA Record
一个完整问答记录，由：
- 1 条用户消息
- 紧随其后的 1 条助手回复

组成。

若存在工具消息、系统消息或多段扩展渲染，则按“逻辑问答单元”进行归并。

### 5.3 Active Window
当前默认保留在 DOM 中的最近消息窗口。

首版默认：**10 组问答记录**。

### 5.4 Placeholder
用于替代真实消息 DOM 的等高占位节点，用于维持滚动高度与定位稳定性。

### 5.5 Snapshot
从真实消息节点提取出的可恢复内容，包含 HTML、文本、尺寸、元信息等。

### 5.6 Restore
将已卸载记录根据 snapshot 重新插回 DOM 的过程。

## 6. 用户体验定义

## 6.1 默认行为

在进入某个长对话页面后：

1. 插件识别当前会话中的问答记录。
2. 默认只保留**最近 10 组问答记录**挂载在 DOM 中。
3. 更早记录会被替换为占位块，不直接显示真实内容。
4. 用户平时浏览最近内容时，不感知旧内容已被卸载。

## 6.2 向上滚动加载

当用户向上滚动，并触发“顶部加载条件”时：

1. 插件从已缓存的旧记录中恢复上一批问答记录。
2. 默认每次加载 5 组问答记录。
3. 新恢复内容插入到最顶部旧占位区域。
4. 恢复后进行滚动补偿，保证视口不突然跳动。

## 6.3 全局搜索行为

当用户使用插件提供的全局搜索时：

1. 插件对当前会话缓存的历史记录做全文搜索。
2. 若命中未挂载的旧记录，则按需恢复：
   - 命中记录本身
   - 其前后少量上下文（例如前后各 1–2 组问答）
3. 视口定位到命中区域。
4. 可在命中区域附近临时扩展 active window。

## 6.4 生成中保护

若最后一组问答仍处于流式生成状态：

1. 该记录不得被卸载。
2. 在内容稳定前，不参与压缩与历史剔除。
3. 稳定后，再纳入普通窗口管理。

## 7. 功能需求

## 7.1 FR-01：会话识别
插件必须能够识别当前页面是否为目标聊天页面，并提取会话上下文。

### 验收标准
- 能在目标站点识别当前会话视图
- 会话切换时能感知 URL 或页面状态变化
- 切换会话后会重建索引，不混用旧会话缓存

## 7.2 FR-02：问答归并
插件必须能将页面上的消息节点归并为“问答记录”。

### 规则
- 默认按 user → assistant 配对
- 中间若存在工具块或补充渲染，则归入所属记录
- 未完成配对的最新用户消息可以作为半成品记录保留

### 验收标准
- 普通对话能正确形成 QA Record
- 不因单条消息的 DOM 细分而错误拆分记录

## 7.3 FR-03：默认保留最近 10 组问答

### 规则
- 初始挂载窗口为最近 10 组 QA Record
- 更早记录可卸载为 placeholder
- 生成中记录固定保留

### 验收标准
- 长对话首次处理完成后，DOM 中真实挂载的记录数接近 10 组问答
- 旧记录被替换为占位块

## 7.4 FR-04：顶部触发按需加载

### 触发条件
满足以下任一条件即可触发：
1. 用户滚动到顶部 sentinel 区域
2. `scrollTop <= topThreshold`
3. 顶部占位区被 IntersectionObserver 命中

### 默认行为
- 每次恢复 5 组更早问答
- 若已经全部恢复，则不再触发

### 验收标准
- 用户滚到顶部后，能逐批恢复旧内容
- 恢复时滚动位置稳定

## 7.5 FR-05：全局搜索

### 范围
- 当前会话全文搜索
- 搜索字段包含：用户消息文本、助手文本、代码块文本、可见标题等

### 行为
- 搜索结果列表显示记录摘要
- 点击某条结果后恢复其内容与上下文
- 若结果已在当前窗口中，则直接定位

### 验收标准
- 未挂载记录可被搜索命中
- 点击后能恢复并跳转到对应位置

## 7.6 FR-06：占位与恢复

### 卸载行为
- 记录序列化为 snapshot
- 记录真实高度写入 placeholder
- 用 placeholder 替换原始 DOM

### 恢复行为
- 靠近视口或搜索命中时恢复
- 恢复后重新测量高度
- 高度变化触发滚动补偿

### 验收标准
- 卸载后页面总高度基本不突变
- 恢复后不出现明显内容错位

## 7.7 FR-07：会话内缓存

### 缓存内容
- QA Record 元数据
- 文本索引
- DOM snapshot
- measured height
- 稳定性状态

### 验收标准
- 当前会话内重复恢复不依赖站点重新渲染旧记录
- 搜索时可直接命中缓存数据

## 7.8 FR-08：页面元素定位辅助

在开发与适配阶段，如需查看网页元素、选择器、滚动容器结构，允许通过 agent 调用 **Playwright** 完成：

- 定位消息列表容器
- 识别用户/助手消息节点结构
- 验证滚动与恢复流程
- 自动采集 DOM 结构变化样本

### 约束
- Playwright 仅作为开发、调试、适配与回归测试工具
- 运行态插件不依赖 Playwright

---

## 8. 非功能需求

## 8.1 性能

1. 长对话页面初始化后，真实挂载记录不应无限增长。
2. 旧记录卸载后，滚动应明显更顺滑。
3. 恢复单批内容时，用户可接受范围内不出现明显冻结。

### 建议指标
- 默认挂载窗口：10 组问答
- 单次顶部恢复：5 组问答
- 预加载缓冲：距视口上下各 1500–2500px
- 批量序列化优先在 idle 阶段执行

## 8.2 稳定性

1. 页面 minor 改版后，不应立即完全失效。
2. 定位策略应尽量避免依赖易变 class 名。
3. 出现未知结构时，插件应自动降级为“不处理”，而不是破坏页面。

## 8.3 可配置性

应至少支持以下配置：
- `windowSizeQa = 10`
- `loadBatchQa = 5`
- `searchContextBefore = 1`
- `searchContextAfter = 1`
- `topThresholdPx`
- `preloadBufferPx`
- `enableSearch`
- `enableVirtualization`

## 8.4 隐私与安全

1. 插件默认不将会话内容上传到外部服务。
2. 搜索与缓存默认仅在本地完成。
3. 调试日志默认关闭。
4. 若后续引入遥测，必须显式开关并默认关闭。

## 9. 系统架构

整体采用四层结构：

1. **Page Adapter**：页面识别与 DOM 定位
2. **Record Engine**：问答归并、索引、状态管理
3. **Virtualization Engine**：卸载、占位、恢复、滚动补偿
4. **Search & UI Layer**：全局搜索、设置面板、调试面板

### 9.1 Page Adapter

职责：
- 定位滚动容器
- 定位消息节点
- 检测会话切换
- 识别页面结构变更

### 9.2 Record Engine

职责：
- 将 turn 聚合为 QA Record
- 分配稳定 ID
- 维护顺序与映射关系
- 维护文本索引与 snapshot 元数据

### 9.3 Virtualization Engine

职责：
- 决定哪些记录挂载
- 决定哪些记录卸载
- 生成 placeholder
- 恢复 snapshot
- 做滚动补偿

### 9.4 Search & UI Layer

职责：
- 提供全局搜索面板
- 展示搜索结果
- 提供开关与配置项
- 提供调试统计

## 11. 核心行为逻辑

## 11.1 初始加载流程

1. 检测当前页面是否是目标聊天页
2. 定位滚动容器
3. 扫描现有消息节点
4. 归并为 QA Record
5. 构建当前会话索引
6. 保留最近 10 组问答
7. 更早记录序列化并替换为 placeholder

## 11.2 顶部触发旧记录加载

1. 监听滚动事件与顶部 sentinel
2. 当触发顶部阈值时，判断是否还有更早记录
3. 取上一批 5 组 QA Record
4. 还原其 DOM
5. 重新计算高度
6. 做滚动补偿，保持用户视口稳定
7. 若超过窗口上限，可根据策略再次压缩更远内容

## 11.3 搜索触发恢复

1. 用户打开插件搜索面板
2. 输入关键词
3. 在本地索引中搜索命中记录
4. 列出命中摘要
5. 用户点击某条结果
6. 恢复目标记录及前后上下文
7. 滚动定位到该记录
8. 暂时扩大附近窗口，防止定位后立即被重新卸载

## 11.4 滚动窗口管理

### 默认规则
- 最近 10 组问答始终优先保留
- 离当前视口较远的已恢复旧记录可以再次压缩
- 搜索命中区域有短期保护期

### 特殊规则
- 生成中的记录强制保留
- 用户正在选中文本的记录不得立即卸载
- 用户悬停交互中的记录应尽量延迟卸载

## 12. DOM 策略

## 12.1 元素定位原则

优先依据以下信息定位：
- 语义标签
- `aria-*` 属性
- `data-*` 属性
- 文本布局特征
- 消息顺序关系

避免强绑定：
- 哈希类名
- 明显易变的构建产物类名
- 深层脆弱 CSS 路径

## 12.2 Placeholder 策略

placeholder 至少包含：
- 固定高度
- recordId
- 是否可恢复标记
- 可选简短摘要（默认隐藏）

示意：

```html
<div class="ecv-placeholder" data-record-id="r-0123" style="height: 824px"></div>
```

## 12.3 Snapshot 策略

首版 snapshot 可采用：
- 根节点 `innerHTML`
- 清洗后的文本内容
- 图片与代码块元信息
- 最终测量高度

首版不要求完全复刻站点内组件行为，只要求恢复后视觉与阅读基本可用。

## 13. 搜索设计

## 13.1 搜索范围

默认仅搜索当前会话缓存。

后续可扩展：
- 所有会话搜索
- 标签搜索
- 代码块优先搜索

## 13.2 索引策略

首版使用轻量本地全文索引即可：
- 直接字符串包含
- 分词匹配
- 简单评分

后续可升级为：
- 倒排索引
- 模糊匹配
- 多字段加权

## 13.3 搜索结果交互

每个结果至少展示：
- 记录序号
- 命中片段摘要
- 命中字段类型
- 点击跳转按钮

## 14. 缓存与存储

## 14.1 内存态

用于运行期：
- 当前会话记录索引
- placeholder 映射
- 当前窗口状态
- 正在生成或保护中的记录集合

## 14.2 持久化

建议：
- 小型配置使用 `chrome.storage.local`
- 大体积 snapshot 与索引使用 IndexedDB

## 14.3 过期策略

首版可采用：
- 按会话隔离
- 仅保留最近 N 个会话缓存
- LRU 淘汰

---

## 15. 关键状态机

## 15.1 Record 状态

记录可处于以下状态之一：

1. `mounted`：真实挂载
2. `serializing`：正在快照化
3. `placeholder`：已卸载，仅保留占位
4. `restoring`：正在恢复
5. `protected`：临时保护，不可卸载
6. `generating`：流式生成中

状态转换示意：

`mounted -> serializing -> placeholder -> restoring -> mounted`

特殊分支：

- `mounted -> protected`
- `mounted -> generating`
- `protected -> mounted`
- `generating -> mounted`

## 16. 错误处理与降级策略

## 16.1 页面结构未知

若无法定位消息容器：
- 插件不执行虚拟化
- 提示“当前页面结构未识别”
- 允许用户提交调试信息

## 16.2 Snapshot 恢复失败

若某条记录恢复失败：
- 标记为坏记录
- 保留 placeholder
- 提示“该条历史恢复失败，可尝试重新扫描”

## 16.3 会话切换中断

若在恢复过程中用户切换会话：
- 中断恢复任务
- 清理当前页面状态
- 重新初始化新会话

## 16.4 页面重渲染冲突

若站点重新插入旧节点：
- 通过 recordId 或结构签名识别重复节点
- 去重
- 必要时重新压缩

## 17. 开发与调试策略

## 17.1 Playwright 使用场景

Playwright 用于开发期，而不是插件运行期。

主要用途：

1. 自动探测滚动容器
2. 采集消息节点 DOM 样本
3. 验证不同会话长度下的结构一致性
4. 回归测试顶部加载与搜索恢复流程
5. 页面改版后快速重新定位选择器策略

## 17.2 调试面板建议项

- 当前 sessionId
- 总 QA Record 数量
- 已挂载记录数
- placeholder 数量
- 最近一次恢复耗时
- 最近一次序列化耗时
- 当前窗口范围
- 是否存在生成中记录

## 19. 里程碑

## M1：结构识别与只读索引

目标：
- 能识别页面
- 能扫描出 QA Record
- 能输出调试信息

## M2：最近 10 组问答窗口

目标：
- 初步实现历史压缩
- 旧记录替换为 placeholder
- 最近 10 组保留

## M3：顶部按批恢复

目标：
- 向上滚动到顶部时恢复更早 5 组
- 基本滚动补偿可用

## 20. 风险与边界

## 20.1 最大风险

1. 站点 DOM 结构频繁变化
2. 原页面框架重新渲染导致节点被回插
3. snapshot 恢复与原样式交互不完全一致
4. 极端长代码块与复杂富文本恢复成本偏高

## 20.2 应对策略

1. 采用语义定位优先，不绑定脆弱 class
2. 使用 Playwright 建立结构回归测试
3. 恢复失败时优先安全降级
4. 保持插件“视觉恢复优先，内部状态一致性次之”的原则

---

## 21. 验收标准

首版完成时，应满足以下最小验收标准：

1. 在目标聊天页面中，能识别并归并问答记录。
2. 默认仅保留最近 10 组问答为真实 DOM。
3. 更早记录能被替换为 placeholder。
4. 滚动到顶部时，能分批恢复更早记录。
5. 全局搜索可命中已卸载的历史记录。
6. 点击搜索结果后，能恢复并跳转到对应位置。
7. 生成中的最后一组问答不会被错误卸载。
8. 页面结构无法识别时，不会破坏原页面。

## 23. 一句话总结

该插件首版不是去重写 ChatGPT 的内部会话系统，而是在网页层实现一个**Gemini 式的历史按需加载与虚拟化阅读体验**：默认只保留最近 10 组问答，旧记录滚到顶部再按批恢复，搜索时再定向恢复相关上下文，从而显著降低长对话页面的卡顿。
